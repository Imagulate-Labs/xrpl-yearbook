# XRP Yearbook — Identity Profiles, NFC Badges & Fair Access

The Yearbook is not just a book. It's a **living community identity system** for the XRP ecosystem: every member gets a profile, a collectible badge, and a place in the history of the chain.

---

## 1. The stack (four layers)

1. **Profile** — an editable public identity page (`xrpyearbook.com/u/<name>`).
2. **Badge** — a physical NFC/QR card that opens the profile with a tap.
3. **Yearbook** — the annual printed + digital archive.
4. **Community Memory** — the long-term record of people, events, projects, and contributions.

The NFC chip is the technology. The badge is the emotion. The profile is the utility. The yearbook is the legacy.

---

## 2. The badge → profile flow

The card stores **one permanent URL** — nothing else. Tap (or scan the QR) → the profile opens.

```
Tap badge → Profile opens → Person is remembered
```

No app, no typing, no searching, no crypto explanation. The card never changes; the profile updates anytime inside the site. One stable URL, a living page behind it.

**Event use case (APEX, hackathons, meetups):**
- *Today:* pull out phone → type name → search X → misspell handle → forget them later.
- *Yearbook:* tap badge → profile opens → follow / save instantly.

---

## 3. Profile pages

**Core fields:** display name · photo · short bio · X / Discord / Telegram · website · project links · community role · yearbook class · event history · badge number · contact.

**Later:** builder / artist / photographer profiles · interview links · media gallery · project timeline · achievements · verified badges · past yearbook appearances.

Feel: a cleaner mix of LinkedIn + Linktree + event badge + yearbook page + community archive.

**Profile modes (same badge, different display):** e.g. *Event Mode* (photography, projects, contact), *Hackathon Mode* (team, project, repo, demo, pitch), *Community Mode*. The NFC URL stays stable; the site controls what shows.

## 4. Badge types

- **Founder** — founding creators / core organizers (e.g. `#001`, Class of 2026).
- **Founding Class** — early members in the first Yearbook.
- **Builder** — developers, founders, project creators.
- **Artist** — visual artists, meme/NFT creators, designers.
- **Community** — active members.

- **Alumni** — appeared in a previous yearbook.

## 5. Physical badge design

Premium, collectible, serious: black base · metallic gold accents · holographic seal · serial number · QR code · embedded NFC · thick PVC/metal card · premium lanyard · clean portrait · logo · badge type + year. Conference credential + luxury collectible + digital identity key. **An artifact, not just access.**

Why people want it: status · identity · belonging · memory · access · recognition · story. People buy the emotion; the NFC is just how it works.

## 6. Fair access (the real problem)

The community keeps getting burned by snipers, bundlers, and buyers grabbing 10–30 items while everyday members miss out. The goal is **not** "sell out fast." It's **fair community distribution.** The question isn't *how do we sell everything?* — it's *how do we make sure the right people get in?*

**Lesson from a prior community drop:** a trusted group enforced one-per-person and slowly redistributed extras **at cost** to people who respected the culture. Result: wide, healthy ownership instead of speculator capture. **Community stewardship works** — and it's worth formalizing.

### Steward allocation (proposed)
- **70%** public / community access
- **15%** team & contributors
- **10%** community-steward reserve (helps real members get in *after* launch, without feeding snipers — transparent, for community health, never secret profit)
- **5%** archive, giveaways, future members

### Anti-sniper rules (proposed)
- One per verified profile in the first claim window.
- No instant unlimited minting; no pure first-come race.
- Profile completion / application required for the first window.
- Public mint opens *after* genuine members get first access.
- Reserve supply for late-discovered real contributors.
- Team avoids dumping; stewards prioritize people who respect the culture.

> Goal isn't perfect prevention — it's **making abuse harder than honest participation.**

### Supply
Not 10,000. Right-size the first edition to feel meaningful, not bloated: **800–2,500** (sweet spots: 1,000 / 1,333 / 1,500). Position it as **membership in a historical moment**, not "just an NFT."

---

## 7. The NFT's role

The NFT is a **receipt / founding-class proof / collectible marker / optional ownership layer** — it *supports* the system, it isn't the product. The real product is the profile + yearbook inclusion + badge + community identity + archive.

---

## 8. Verification

Each badge has a verify endpoint: `xrpyearbook.com/verify/<badge-id>` → shows badge ID · type · owner display name · class · issue date · status. Statuses (human language): **Active · Archived · Revoked · Lost · Transferred.** Can later connect to verifiable receipts.

**Language rule:** keep it human (Active, Verified, Class of 2026, Founding Member). Avoid the heavy crypto jargon (DID, VC, soulbound, sybil-resistant) in the UI — the tech can live underneath, the words stay warm.

> Note: this is **community identity**, not legal identity. It answers *who is this person in the community, what have they built, what events did they attend.*

---

## 9. MVP — what to build, in order

**Build:**
- **Phase 1 — Profiles:** editable public profile pages + public URL.
- **Phase 2 — Badge URLs:** stable URLs + QR codes (NFC-ready).
- **Phase 3 — Admin review:** approve Founding Class, assign badge numbers/types, reserve steward supply, mark status.
- **Phase 4 — Badge print export:** printable badge data.
- **Phase 5 — NFC test:** write a URL to a test card.
- **Phase 6 — Prototype:** Badge `#001`.
- **Phase 7 — Steward allocation:** fair-access tracking.

**Do NOT build yet:** full social network · news feed · token marketplace · complex wallet gating · on-chain reputation · full DID system · heavy gamification · complex contracts.

> Start with the **human workflow**, not blockchain. *Create profile → get profile URL → badge opens profile.*

## 11. Guiding principle

Lead by example; don't add to bad crypto behavior. **Fair access · community first · real utility · physical + digital memory · anti-vulture distribution · long-term culture · human-first design · preserve the people who built the community.**

> The technology should **protect** the community, not exploit it.
